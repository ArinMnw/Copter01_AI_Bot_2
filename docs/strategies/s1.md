## แนวคิด

- ใช้แท่ง `[2]`, `[1]`, `[0]` (เก่า→ใหม่ ปิดสมบูรณ์แล้วทั้งหมด) เป็นแกนหา pattern กลืนกิน/ตำหนิ — Pattern F ใช้แค่ `[1]`,`[0]` (2 แท่ง), Pattern G ใช้ `[3]`..`[0]` (4 แท่ง)
- มีทั้งฝั่ง BUY และ SELL (สลับสีตรงข้ามกันทุก pattern)
- ใช้ `engulf_min_price()` เป็นระยะขั้นต่ำของคำว่า "กลืนกิน" (gap) — scale ตาม instrument
- body ขั้นต่ำของแท่งที่เช็ค (ตามแต่ pattern) = `35%`
- รองรับ **Zone Mode** (`S1_ZONE_MODE = "zone"`) และ **Forward Confirm** เพิ่มเติม
- ชื่อ pattern A-G ในเอกสารนี้คือชื่อไล่ตัวอักษรเพื่ออ่านง่าย — **ไม่ตรงกับเลข comment code ใน MT5** (เช่น G ใช้ comment `P4`, F ใช้ `P6`) เพราะ comment code ถูกกำหนดไว้ก่อนที่ G จะถูกเพิ่มเข้ามาทีหลัง

## Pattern (เรียงตามตัวอักษร A→G)

ทุก pattern ด้านล่างเขียนเฉพาะฝั่ง **BUY** — ฝั่ง **SELL สลับสีตรงข้ามทั้งหมด** (เขียว↔แดง, High↔Low, ทิศทาง entry กลับด้าน)

### Pattern A — กลืนกิน (`P1`)

```
[2] แดง
[1] เขียว, Close[1] > High[2] + gap        ← กลืน [2]
[0] เขียว, Close[0] > High[1] + gap        ← กลืน [1] (ยืนยัน)
```
เงื่อนไขเพิ่ม: body ของ `[1]` ≥ 35%
Entry = Fibo 50% ของ High-Low[1] | SL = min(Low[0..2]) − buffer | TP = Swing High ถัดไป (ไม่เจอ → RR 1:1)

### Pattern B — ตำหนิ (`P2`)

```
[2] แดง
[1] เขียว "ตำหนิ" (High[1] ≥ Open[2], ไม่ต้องปิดเหนือ High[2] เต็มแบบ A)
[0] เขียว, Close[0] > High[1] + gap        ← กลืน [1] (ยืนยัน)
```
เงื่อนไขเพิ่ม: body ของ `[1]` ≥ 35%
Entry/SL/TP สูตรเดียวกับ Pattern A

### Pattern C — ย้อนโครงสร้าง (`P3`)

```
[2] แดง
[1] เขียว (ไม่เช็คความสัมพันธ์กับ [2] เลย — หลวมที่สุดในกลุ่ม)
[0] เขียว, Close[0] > High[1] + gap        ← กลืน [1] (ยืนยัน)
```
เงื่อนไขเพิ่ม: body ของ `[1]` ≥ 35%
Entry/SL/TP สูตรเดียวกับ Pattern A/B

### Pattern E — กลืนกิน 2 แดง (`P5`)

```
[2] แดง
[1] แดง (สีเดียวกับ [2] — ต่างจาก A/B/C ที่ [1] ต้องเป็นเขียว)
[0] เขียว, Close[0] > High[1] + gap, body[0] ≥ 35%   ← กลืน [1] เดียวจบ
```
Entry = Fibo 50% ของ Body **[0]** (ไม่ใช่ `[1]` เหมือน A/B/C) | SL = min(Low[0..2]) − buffer | TP = Swing High
`cancel_bars = 0` (เดิม `1` — ยกเลิกหลัง 1 แท่งถ้าไม่ fill, เอาออกแล้ว 2026-07-14 หลัง backtest พบว่า pattern นี้ไม่ใช่ตัวที่ทำให้ผลแย่ลง)

### Pattern F — 2 แท่งกลืนกิน (`P6`)

```
[1] แดง
[0] เขียว, Close[0] > High[1] + gap        ← กลืน [1]
```
สั้นสุด — ไม่มีแท่ง `[2]` เลย ไม่เช็ค body ขั้นต่ำของ `[1]`
Entry = **Open[1]** (ไม่ใช่ Fibo 50% เหมือนตัวอื่น) | SL = max(High[0],High[1]) + buffer | TP = Swing High
`cancel_bars = 0` (เดิม `1`, เอาออกพร้อม Pattern E)

### Pattern G — 4 แท่ง (`P4`)

```
[3] แดง
[2] เขียว, body ≥ 35%                       ← แท่งกลืนหลัก
[1] เขียว, Close[1] < High[2]               ← พักตัว ยังเขียวแต่ไม่ทะลุ High ของ [2]
[0] เขียว, Close[0] ≥ High[1]               ← กลืน [1] ยืนยันไปต่อ
```
Entry = Fibo 50% ของ Body **[2]** (แท่งกลืนหลัก ไม่ใช่แท่งท้ายสุด) | SL = min(Low[0..3]) − buffer | TP = Swing High
ระยะรอ fill มักไกลกว่า pattern อื่นเพราะ entry ย้อนกลับไปที่แท่งกลืนหลัก ไม่ใช่แท่งล่าสุด

## Zone Mode (`S1_ZONE_MODE = "zone"`)

- ขั้น detect จะ **ไม่ block** setup เพราะหลุด zone — ตั้ง order ไปก่อน แล้วประเมินทีหลัง
- การเช็ก zone ใช้ `min low` (BUY) / `max high` (SELL) ของ **ทุกแท่งใน pattern** เทียบ Swing Low/High (ไม่ใช่แค่แท่ง `[1]`)
- pending อยู่นอก zone → ยกเลิก limit
- pending ยังอยู่ใน zone → คง order
- fill แล้วอยู่นอก zone และ `profit < 0` → ปิด position
- fill แล้วอยู่นอก zone แต่ `profit >= 0` → ไม่ปิด
- metadata แนบกับ setup ใน `s1_zone_meta`

## Forward Confirm

- S1 ตั้ง order ไปก่อน แล้วรอดู S2 หรือ S3 ฝั่งเดียวกันใน TF เดียวกันภายใน **5 แท่งข้างหน้า**
- ยัง pending + ครบ 5 แท่งไม่เจอ → ยกเลิก pending
- fill แล้ว + ครบ 5 แท่งไม่เจอ → ปิด position
- เจอแล้ว (ไม่ว่า pending หรือ fill) → ไม่ทำอะไรต่อ

## Entry / SL / TP

- entry/SL/TP คำนวณจาก pattern + swing (ดู `entry_calculator.py` / `tp_sl/strategy1_tp_sl.py`)
- TP พยายามหา swing TP ที่เหมาะสม → ถ้าไม่เจอใช้ fallback **RR 1:1**
