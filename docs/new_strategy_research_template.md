# New Strategy Research Template (standalone, backtest-only)

เอกสารนี้คือ "กฎกลาง" สำหรับงานวิจัย/optimize กลยุทธ์ XAUUSD ใหม่แบบ standalone
(ไม่ wire เข้า live bot) เป้าหมายคือกำไรเฉลี่ย **$1000 ต่อวัน** จากทุนเริ่มต้น **$1000**

ทุกครั้งที่ใช้ `/loop` สำหรับงานวิจัยกลยุทธ์ใหม่ ให้ "อ่านไฟล์นี้ก่อน" แล้วทำตามกฎด้านล่างทั้งหมด
ไม่ต้อง copy กฎไปแปะใน prompt ซ้ำ — prompt แค่บอกให้มาอ่านไฟล์นี้แทน

## Scope และข้อจำกัด

- ทุนเริ่มต้น backtest: $1000 (ไม่เกี่ยวกับเงินฝากจริง)
- เป้าหมาย: กำไรเฉลี่ย **ต่อวัน** (ไม่ใช่ต่อเดือน/สัปดาห์) >= $1000 อย่างสม่ำเสมอ (ไม่ใช่ peak วันเดียว)
- lot/risk ปรับได้ตามเหตุผล แต่ต้องระบุ risk% ต่อไม้ทุกครั้งที่เปลี่ยน และต้องสมเหตุสมผลกับทุน $1000
- รันแบ็คเทสต์จริงด้วยข้อมูล MT5 ในโปรเจกต์เท่านั้น (chart=UTC+6, fetch ต้องบวก +1h) ห้ามมั่วผลลัพธ์
- ห้ามแก้ไฟล์กลยุทธ์เดิม (S1-S20) หรือไฟล์ระบบหลัก (bot_state.json, supervisor, main bot loop,
  state machine กลาง) ห้าม wire กลยุทธ์ใหม่เข้า live trading จนกว่าผู้ใช้สั่งแยกต่างหาก
- ทุกกลยุทธ์ใหม่อยู่ในสถานะ research/backtest-only 100%

## กฎข้อที่ 1 — Lock อยู่กับกลยุทธ์เดียว ห้ามงอกตัวใหม่ง่ายๆ

- ก่อนเริ่ม ให้เช็คเลขกลยุทธ์ล่าสุดในโปรเจกต์ แล้วใช้เลขถัดไปที่ยังไม่ถูกใช้ 1 เลขเท่านั้น
  (เช่น strategy21.py) — **ใช้เลขนี้เลขเดียวตลอดทั้ง session/loop**
- สร้าง `strategy<N>.py` (ตาม pattern strategy1.py-strategy20.py) และ `sim_s<N>_backtest.py`
  (ตาม pattern sim_s1_backtest.py-sim_s20_backtest.py)
- การ "พัฒนากลยุทธ์ให้ดีขึ้น" ต้องทำโดยการ**เพิ่มความลึกให้กลยุทธ์ตัวเดิม** เช่น:
  - เพิ่ม confirmation filter ใหม่ (volume, multi-timeframe confluence, momentum, ADX/ATR regime)
  - ปรับ exit logic ให้ฉลาดขึ้น (partial TP, trailing, time-based exit, breakeven)
  - ปรับ position sizing ให้ฉลาดขึ้น (Kelly fraction, scale-in/out ตาม signal strength)
  - เพิ่ม session/regime filter ที่มีเหตุผลรองรับจากข้อมูลจริง (ไม่ใช่สุ่ม)
  ไม่ใช่การไปออกแบบกลยุทธ์ใหม่อีกตัวที่ไม่เกี่ยวข้องกัน
- ห้ามขึ้นเลขกลยุทธ์ใหม่ จนกว่าจะผ่านเงื่อนไข "Exhaustion Checklist" ข้อ 4 ด้านล่างครบทุกข้อ
  และมีการบันทึกเหตุผลไว้ใน `create_s<N>.md` แล้วเท่านั้น

## กฎข้อที่ 2 — Grid Search อัตโนมัติ ไม่ใช่ tune มือสุ่มๆ

- ก่อนเริ่ม tune มือ ให้เขียน grid search script (เช่น `optimize_s<N>.py` หรือใช้
  `optimize_backtest.py` เดิมเป็นฐาน) ที่ลูปพารามิเตอร์หลักทั้งหมดแบบมีระบบ ครอบคลุมอย่างน้อย:
  entry filter/threshold, SL/TP ratio, session window, risk%, timeframe/confirmation
- รันให้ครอบคลุมพารามิเตอร์อย่างน้อย **50 combination ที่มีความหมาย** ก่อนจะสรุปผลใดๆ
  เกี่ยวกับ "ขอบ" (edge) ของกลยุทธ์นี้
- ห้ามเลือก "ดีที่สุดในรอบที่ลองมา" แบบไม่ครอบคลุม space ที่กำหนดไว้ก่อน

## กฎข้อที่ 3 — แยก "leverage scaling" ออกจาก "edge ที่ดีขึ้นจริง"

- ทุกครั้งที่ปรับ risk%/lot ให้รายงานคู่กันเสมอ: `avg_per_day` และ `max_dd_pct` ที่ risk นั้น
- ถ้า `avg_per_day` กับ `max_dd_pct` โตในสัดส่วนใกล้เคียงกันเมื่อเพิ่ม risk% (เช่น risk เพิ่ม 10x,
  ทั้งสองค่าโตประมาณ 10x เหมือนกัน) ให้ระบุชัดว่านี่คือ **"leverage scaling ไม่ใช่ edge improvement"**
  แล้วกลับไปหา edge จริงต่อ (กฎข้อ 1) ไม่ใช่ใช้ผลนี้เป็นคำตอบสุดท้าย
- หน่วยวัด "edge จริง" ที่ต้องรายงานคู่กับทุกรอบ optimize: expectancy ต่อไม้ (R-multiple เฉลี่ย),
  win rate, profit factor — เพราะค่าพวกนี้ไม่เปลี่ยนตาม risk% ที่ตั้ง ต่างจาก `avg_per_day` ดอลลาร์ดิบ
- **Fixed-lot sanity check (บังคับถ้า trade frequency > ~5 ไม้/วัน — ปรับลดจาก 10 หลังพบว่า S54 ที่
  7.3 ไม้/วันก็บิดเบือนมีนัยสำคัญแล้ว):** position sizing แบบ
  `risk_usd = equity x risk_pct` (เห็นใน `sim_s31_backtest.py::simulate_equity_substream`) ผสมกับ
  ความถี่เทรดสูงมาก สามารถสร้าง PF ปลอมขึ้นมาเองทางคณิตศาสตร์จาก lot-rounding ผสม equity ที่โตเร็ว
  (พบจริงใน S53: compounding sim รายงาน PF=1.55-1.63 แต่คำนวณที่ fixed lot 0.01 คงที่ตลอด — ไม่ผ่าน
  compounding เลย — ได้ PF จริงแค่ 0.98-1.09 คือไม่มี edge เลย) ก่อนเชื่อ PF/sharpe ของ leg ความถี่
  สูงใดๆ ต้อง sum `diff_usd_per_001lot` ตรงๆ (ไม่ผ่าน `simulate_equity_substream`) แล้วเทียบ PF ที่
  ได้กับ PF จาก compounding sim — ถ้าต่างกันมาก ให้เชื่อค่า fixed-lot เป็นค่า PF จริงของกลไก

## กฎข้อที่ 4 — Exhaustion Checklist (ต้องผ่านครบก่อนสรุปว่า "ทำไม่ได้" หรือขึ้นเลขใหม่)

ก่อนประกาศว่ากลยุทธ์ปัจจุบัน "ทำไม่ได้" หรือก่อนขึ้นเลขกลยุทธ์ใหม่ ต้องผ่านครบทุกข้อ:

1. [ ] รัน grid search อย่างน้อย 50 combination ตามกฎข้อ 2 แล้ว
2. [ ] ลอง edge-improvement อย่างน้อย 2 แนวทางที่ต่างกันโดยสิ้นเชิง (เช่น เพิ่ม filter ประเภท A
       แล้วลองเพิ่ม filter ประเภท B ที่ไม่เกี่ยวกัน) ไม่ใช่แค่ขยับตัวเลขพารามิเตอร์เดิม
3. [ ] ทำ sanity-check โค้ด: print รายละเอียด trade ตัวอย่าง 5-10 ไม้ (entry, SL, TP, direction)
       แล้วยืนยันด้วยตาว่าตรรกะถูกต้อง (ไม่มีบั๊กแบบ S24 ที่ SL วางผิดฝั่ง) — ถ้าเจอบั๊กต้องแก้แล้ว
       รันใหม่ก่อน ไม่ใช่ทิ้งไปขึ้นตัวใหม่
4. [ ] คำนวณ "expectancy ที่ต้องการ" เทียบกับ "expectancy ที่หาได้จริงจากข้อมูล" แบบมีตัวเลข
       (เช่น ด้วยความถี่ไม้ที่ทำได้จริง ต้องมี R-multiple เฉลี่ยเท่าไหร่ถึงจะถึง $1000/วันที่ risk
       ระดับปลอดภัย (DD <= 20-25% ของทุน) แล้วเทียบกับ R-multiple เฉลี่ยที่กลยุทธ์ทำได้จริง)
5. [ ] เขียนสรุปทั้งหมดนี้ลงท้าย `create_s<N>.md` ก่อน ถ้าไม่มีสรุปนี้ห้ามขึ้นเลขใหม่หรือเลิกงาน

## กฎข้อที่ 5 — ห้ามหยุด loop เพื่อรอคำสั่งผู้ใช้

- `/loop` ต้องทำงานต่อเนื่องด้วยตัวเอง ห้ามจบ session ด้วยการ "รอคำสั่งผู้ใช้ว่าจะทำอะไรต่อ"
- ถ้าผ่าน Exhaustion Checklist ครบแล้วและสรุปว่าเป้าหมาย $1000/วันที่ risk ปลอดภัยทำไม่ได้จริง
  (มีตัวเลขรองรับตามข้อ 4) ให้ทำสิ่งนี้แทนการหยุดรอ:
  1. เลือก config ที่ risk-adjusted ดีที่สุดที่ทำได้จริงเป็นคำตอบสุดท้ายของกลยุทธ์นี้
  2. บันทึกเป็นข้อสรุปสุดท้ายใน `create_s<N>.md` พร้อมตัวเลขเปรียบเทียบเป้าหมาย vs ผลจริง
  3. จบ loop ด้วยรายงานสรุปสุดท้ายที่ชัดเจน ไม่ใช่คำถามเปิดให้ผู้ใช้ตัดสินใจ

## ไฟล์บันทึกผล (ต้องมีครบ และอัพเดททุกรอบ ไม่ใช่เขียนทีเดียวตอนจบ)

- `s<N>_backtest_summary.csv` — log ตัวเลขทุกรอบ optimize (timestamp, label, params, avg_per_day,
  win rate, max_dd, risk%, expectancy/R-multiple)
- `create_s<N>.md` — บันทึกเป็นข้อความอ่านง่าย: ไอเดียกลยุทธ์/แหล่งที่มา, เหตุผลที่เลือกทำตัวนี้,
  รายการทุกรอบ optimize + ผลลัพธ์, บั๊กที่เจอและวิธีแก้, สถานะ exhaustion checklist, บทสรุปสุดท้าย

## Definition of Done

ถือว่างานวิจัยกลยุทธ์นี้ "เสร็จ" เมื่อข้อใดข้อหนึ่งเป็นจริง:

- (ก) backtest ได้กำไรเฉลี่ย >= $1000/วันอย่างสม่ำเสมอ โดย max drawdown สมเหตุสมผล
  (ไม่ใช่มาจาก leverage scaling ตามกฎข้อ 3) — รายงานพร้อมพารามิเตอร์สุดท้าย
- (ข) ผ่าน Exhaustion Checklist ครบทุกข้อ (กฎข้อ 4) และพิสูจน์ได้ด้วยตัวเลขว่าเป้าหมายนี้ทำไม่ได้
  ที่ risk ระดับปลอดภัย — รายงาน config ที่ดีที่สุดที่ทำได้จริงแทน พร้อมเหตุผลเชิงตัวเลขครบถ้วน
