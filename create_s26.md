## สมมติฐานที่ทดสอบ

S21-S25 ทุกตัวความถี่ต่ำ (2-6 ไม้/วัน) แม้บางตัวมี edge บวก (avgR เป็นบวก) ก็ไปไม่ถึง
$1000/วันที่ risk ปลอดภัย เพราะไม้น้อยเกินไป (ดู `create_s25.md` — ขาดอยู่ ~50-330 เท่า)
S26 ทดสอบสมมติฐานตรงข้าม: ถ้าลง timeframe ไปที่ M1 (ละเอียดสุดที่มีข้อมูล MT5) ด้วย "ท่าเดียวที่
เทรดซ้ำได้บ่อยมาก" + RR1:1 fixed (ง่ายต่อการ hit TP แต่ต้องการ WR>50%+spread cost ถึงจะ breakeven)
จะปิดช่องว่างไปเป้าได้ด้วยความถี่สูงแทนหรือไม่

## ไอเดียกลยุทธ์ / แหล่งที่มา — ลอง 3 ท่าในกริดเดียวกัน (ใส่เป็น `SETUP_TYPE`)

1. **ema_pullback** — ราคา M1 แตะ EMA fast แล้วเด้งตามเทรนด์ EMA_TREND (M1 เอง ไม่ใช่ M5/M15
   เพื่อให้ signal ถี่พอ — ทดสอบ EMA_TREND ทั้ง 21 และ 50)
2. **momentum_pullback** — แท่ง momentum แรง (body >= mult×ATR) ตามด้วยแท่ง pullback เล็ก
   แล้วเข้าตามทิศ momentum (ไม่รอ retest กลับไปแนวรับ/แนวต้านแบบ order-block เต็มรูปแบบ
   — ปรับให้เกิดถี่กว่าเดิมตามจุดประสงค์ความถี่สูง)
3. **range_scalp** — fade ที่ขอบกรอบ range (lookback N แท่ง) ช่วง session ผันผวนสูง
   (London/NY) ด้วย rejection candle (ปิดสวนทิศที่แตะขอบ)

(ตัดท่า "micro order-block retest" ออกจากการทดสอบเชิงลึก เพราะ logic ใกล้เคียง range_scalp
มากเกินไปในกรอบ M1 ที่ noise สูง — ใช้ 3 ท่าที่ต่างกันมากกว่าแทนตามขอบเขตที่ระบุไว้)

ไฟล์: `strategy26.py` (รวม logic 3 ท่า เลือกด้วย `S26_DEFAULTS["SETUP_TYPE"]`) /
`sim_s26_backtest.py` (backtest M1 จาก MT5 จริง) / `optimize_s26.py` (grid search)

### ข้อแตกต่างจาก S21-S25 (กฎข้อ 1)
ทุกท่าใน S26 อยู่บน **M1** (ไม่ใช่ M5/M15/H1 เหมือน S21-S25 ทั้งหมด) และใช้ **RR fixed 1:1**
(S21-S25 ใช้ RR 1.5-2.5) — กลไกที่ทดสอบคือ "ความถี่สูงมาก" ไม่ใช่ "WR สูงต่อไม้" เหมือนกลุ่มก่อน

## Entry/Exit (ทุกท่าเหมือนกัน)

- Entry: MARKET ที่ open ของแท่งถัดจากแท่ง signal (กัน look-ahead)
- SL: ตาม logic เฉพาะท่า + ATR(14,M1) buffer (`SL_ATR_MULT`)
- TP: entry ∓ 1.0×risk (**TP_RR fixed = 1.0 ตามที่ผู้ใช้สั่ง — ไม่ปรับค่านี้ในกริดทั้งหมด**)
- Position sizing: risk-based ตาม % equity ต่อไม้ (เหมือน S21-S25)
- กันสัญญาณซ้อนทับโดยตรง: ห้ามยิงสัญญาณใหม่ภายใน 2 แท่งหลังสัญญาณก่อนหน้า (`min_gap_bars=2`
  ใน `sim_s26_backtest.py`)

**สรุปกฎข้อ 2 (Exhaustion Checklist):** ทดสอบ edge-improvement 2 แนวทางที่ต่างกันโดยสิ้นเชิง

## ข้อ 4 — คำนวณ expectancy ที่ต้องการ vs ที่ทำได้จริง (ตัวเลข)

### สรุป WR และความถี่จริงที่หาได้จากข้อมูล MT5 จริง (ไม่ประมาณ)

- **WR สูงสุดที่หาได้จริง (robust, ไม่ใช่ overfitting):** ~52.6-53.3% (range_scalp,
  locked config / London-core session) — ค่าที่ดูสูงกว่านี้ (53.6% ที่ combo 30วัน) พิสูจน์แล้ว
  ว่าเป็น noise จาก sample เล็ก (ตกลงมาที่ 52.1% บน 60 วัน)
- **WR สูงสุดที่หาได้จริงทั้งกริด 108+ combination:** 53.6% (เกิดเฉพาะ sample เล็กที่ overfit)
  ไม่มี combination ใดถึง 55% แบบ robust ตลอดทั้งกริด
- **ความถี่ไม้/วันจริงสูงสุดที่ data จริงเอื้อให้เกิด (ทุก setup_type รวม):** **140.4 ไม้/วัน
  (active)** (range_scalp, lb=10, edge=0.25 — แต่ WR ที่ความถี่นี้เหลือเพียง 50.1-50.6%)
- **ความถี่ไม้/วันจริงของ locked config ที่ WR ดีที่สุด:** 92.7 ไม้/วัน (full session) หรือ
  41.5 ไม้/วัน (London-core เท่านั้น ที่ WR ดีขึ้นเป็น 53.3%)

**ข้อสรุปสำคัญ:** มี **trade-off ตรงข้ามกันชัดเจน** ระหว่าง WR กับความถี่ — filter ที่เพิ่ม WR
(narrow session, ATR regime) ลดความถี่ลง 50-90% เสมอ ไม่มี combination ใดที่ได้ทั้ง WR สูง
**และ** ความถี่สูงพร้อมกัน

### เทียบกับเกณฑ์ที่ผู้ใช้กำหนด (WR ต้อง >= 65% ถึงจะ "ผ่าน")

**WR จริงสูงสุดที่หาได้ (52.6-53.6%) ต่ำกว่าเกณฑ์ 65% ที่กำหนดไว้อย่างชัดเจน (ขาดอยู่ ~11.4-12.4
percentage points)** → **ตาม "ห้ามดัน lot/risk แทนการแก้ WR" แนวทางนี้ถือว่า "ไม่ผ่าน" ตั้งแต่
ขั้นนี้แล้ว** ไม่ต้องพิจารณาต่อว่าความถี่พอหรือไม่ (แต่ตรวจสอบเพิ่มเพื่อความสมบูรณ์ของ
Exhaustion Checklist)

เลยด้วยซ้ำ — กล่าวคือ S26 ไม้น้อยกว่าที่ตารางนี้ต้องการ "ที่ WR ต่ำสุดในตาราง" และ WR ก็ต่ำกว่า
แถวต่ำสุดในตารางนั้นไปอีก **ขาดทั้ง 2 มิติพร้อมกัน** ไม่ใช่ขาดมิติเดียว

### ข้อค้นพบเชิงโครงสร้างที่สำคัญกว่า — เพดาน R ของ RR1:1

เพราะ TP_RR fixed = 1.0 **เพดานบนสุดของ avgR ต่อไม้ (ที่ WR=100% ทุกไม้ ซึ่งเป็นไปไม่ได้จริง)
คือ +1.0R เท่านั้น (ลบด้วย spread cost เล็กน้อย)** ที่ locked config (92.7 ไม้/วัน, risk=1%
=$10/ไม้บนทุน $1000): ต้องการ avgR ต่อไม้ = ($1000/92.7) ÷ $10 = **+1.078 R/ไม้**

**+1.078R ต่อไม้ "เกินเพดานทางทฤษฎีของ RR1:1" (≈+1.0R สูงสุด) อยู่แล้ว ไม่ว่า WR จะสูงแค่ไหนก็
ทำไม่ได้ที่ความถี่นี้และ risk 1%** — นี่คือข้อจำกัดทางโครงสร้าง (structural ceiling) ของการล็อก
RR ไว้ที่ 1:1 ไม่ใช่แค่ปัญหา edge ไม่พอ ต้องเพิ่มความถี่ให้เกิน ~107 ไม้/วัน (ที่ WR=100% สมมุติ
เท่านั้น) จึงจะมีโอกาสในทางทฤษฎี ส่วนที่ WR จริง 52.6% avgR เพดานจริงคือ
2×0.526-1-spread_R ≈ **+0.0 ถึง -0.02R เท่านั้น** (ตรงกับที่วัดได้จริง -0.016R)

**ส่วนต่าง (expectancy gap) ที่ locked config:** ต้องการ +1.078R, ทำได้จริง -0.016R
**= ขาดอยู่ ~1.094R ต่อไม้ (ติดลบทั้งหมด ไม่ใช่แค่ไม่พอ — ทิศทางผิดด้วย)** ต่อให้ดัน risk%
ขึ้นก็ไม่ช่วย เพราะ avgR ไม่ขึ้นกับ risk% (ยืนยันจากตารางกฎข้อ 3 ด้านบน)
