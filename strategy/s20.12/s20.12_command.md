# S20.12 Command Reference

หน้านี้รวบรวมคำสั่งสำหรับใช้งานสคริปต์ Backtest และ Compare ของกลยุทธ์ `S20.12 FutureKey` (Candlestick Mechanics) ครับ คุณสามารถก๊อปปี้คำสั่งในกรอบสีดำไปรันที่ Terminal ได้เลย

---

### 0. เลือกบัญชีด้วย `BOT_PROFILE` (สำคัญ — เช็คก่อนรันทุกครั้ง)

ทั้ง `backtest_S20_12_runner_mt5.py` และ `compare_mt5_orders.py` login เข้าบัญชีตาม env var `BOT_PROFILE` — ถ้า**ไม่ได้ตั้งไว้**จะ fallback ไปใช้ค่า default ใน `config.py` (บัญชี demo `2101114448` @ IUXMarkets-Demo) โดยอัตโนมัติ

**รันแบบไม่ระบุ profile (ใช้ default demo):**
```bash
python strategy/s20.12/backtest-sim/backtest_S20_12_runner_mt5.py --days 30 --tf all --compound 2
```

**รันแบบระบุ profile ชัดเจน (ตั้ง env var ก่อนรัน):**
```bash
BOT_PROFILE=demo-iux-2101114448 python strategy/s20.12/backtest-sim/backtest_S20_12_runner_mt5.py --days 30 --tf all --compound 2
```
```bash
BOT_PROFILE=real-vtmarket-26575165 python strategy/s20.12/backtest-sim/compare_mt5_orders.py
```

- ชื่อ profile ที่มีอยู่ตอนนี้ดูได้จาก `profiles/demo/` และ `profiles/real/` (เช่น `demo-iux-2101114448`, `real-vtmarket-26575165`)
- ตั้ง `BOT_PROFILE` ผิด/ไม่ตรงกับบัญชีที่ live bot รันอยู่จริง จะทำให้ `compare_mt5_orders.py` ดึงประวัติออเดอร์จากคนละบัญชี → เทียบไม่ได้เลยแม้ backtest จะถูกต้อง
- ถ้าจะเทียบกับ live bot ที่กำลังรันอยู่ ต้องใช้ `BOT_PROFILE` **ตัวเดียวกัน**กับที่ตั้งไว้ตอนสตาร์ทบอทจริงเสมอ

### 1. `backtest_S20_12_runner_mt5.py`
สคริปต์ Backtest พื้นฐานที่ดึงข้อมูลจาก MT5 โดยตรง จำลอง order แบบ market (เข้าทันทีตอนเจอ pattern ไม่รอ limit fill — ตรงกับ live bot จริง) พร้อม portfolio simulation แบบ event-driven (รวมทุก TF เรียงตามเวลาเปิด/ปิดจริง เริ่ม balance ที่ $1,000 เสมอ)

**คำสั่งรันแบบมาตรฐาน (แนะนำให้บันทึก log):**
```bash
python -u strategy/s20.12/backtest-sim/backtest_S20_12_runner_mt5.py --days 30 --tf all --compound 2 > bt_s20_12_out.log 2>&1
```
*(คำสั่งนี้จะทำการเทส 30 วัน, สแกนทุก Timeframe, ทบต้นที่ 2% และบันทึกผลลัพธ์ไว้ในไฟล์ `bt_s20_12_out.log`)*

**คำสั่งรันแบบดูผลลัพธ์บนจอโดยตรง:**
```bash
python strategy/s20.12/backtest-sim/backtest_S20_12_runner_mt5.py --days 30 --tf all --compound 2
```

**คำสั่งรันแบบระบุเวลาเริ่มต้นชัดเจน (จากเวลานั้นจนถึงปัจจุบัน):**
```bash
python strategy/s20.12/backtest-sim/backtest_S20_12_runner_mt5.py --start "02-07-2026 18:00" --tf all --compound 2
```

**คำสั่งรันแบบระบุช่วงเวลาตายตัวทั้งต้น-ท้าย:**
```bash
python strategy/s20.12/backtest-sim/backtest_S20_12_runner_mt5.py --start "02-07-2026 18:00" --end "02-07-2026 22:00" --tf all --compound 2
```

*ตัวอย่างพารามิเตอร์เพิ่มเติม:*
- `--days 60` (ระบุจำนวนวันที่ต้องการเทสย้อนหลัง — ถ้าไม่ระบุทั้ง `--days` และ `--start` จะรันวนหลายช่วง 30/60/90/120/180 วันอัตโนมัติ)
- `--start "dd-MM-yyyy HH:mm"` (ระบุเวลาเริ่มต้นแบบ BKK ตรงๆ วิ่งจากเวลานั้นถึงปัจจุบัน (หรือถึง `--end` ถ้าระบุ) — ถ้าใส่พารามิเตอร์นี้จะ override `--days`)
- `--end "dd-MM-yyyy HH:mm"` (ระบุเวลาสิ้นสุดแบบ BKK ตรงๆ — ถ้าไม่ระบุ = ปัจจุบัน ใช้ร่วมกับ `--start` เพื่อกำหนดช่วงตายตัว หรือใช้ร่วมกับ `--days` เพื่อนับถอยหลัง N วันจาก `--end` แทนที่จะนับจากปัจจุบัน)
- `--tf M15` (ระบุ Timeframe เจาะจง เช่น M1, M5, H1, D1 หรือ all)
- `--compound 1.0` (ระบุเปอร์เซ็นต์ความเสี่ยง/พอร์ต สำหรับคำนวณการทบต้น Lot Size)

ผลลัพธ์จะถูกบันทึกไว้ที่ `strategy/s20.12/excel/s20_12_sim_trades.csv` (คอลัมน์: Time (BKK), Close Time, TF, Type, Entry, SL, TP, **Lot**, P&L, **Balance**, Reason) เรียงตามเวลาเปิดจริง

### 2. `compare_mt5_orders.py`
เทียบออเดอร์จาก SIM (`s20_12_sim_trades.csv`) กับออเดอร์จริงบน MT5 — ต้องรัน `backtest_S20_12_runner_mt5.py` ก่อนเสมอเพื่อสร้างไฟล์ SIM ให้ตรงกับช่วงเวลาที่ต้องการเทียบ

```bash
python strategy/s20.12/backtest-sim/compare_mt5_orders.py
```

**ถ้ารัน backtest ด้วย `--start`/`--end` มา ต้องใส่ค่าเดียวกันให้ compare ด้วยเสมอ** (ไม่งั้น order กำพร้าที่อยู่นอกช่วงจริงจะหลุดเข้ามาปน หรือ order ท้ายช่วงที่ควรอยู่ในขอบเขตจะถูกตัดออกไป):
```bash
python strategy/s20.12/backtest-sim/compare_mt5_orders.py --start "02-07-2026 18:00" --end "02-07-2026 22:00"
```

- Match กันเมื่อเวลา **open และ close ตรงกันระดับ ชม:นาที** (ไม่สนวินาที) เท่านั้น ทั้งสองฝั่ง
- ออเดอร์จริงที่ไม่มี SIM คู่เลย (backtest ไม่เจอ pattern เดียวกัน) จะโชว์เป็นแถวแยกพร้อม note "ไม่มี SIM คู่" — กรองด้วยช่วงเวลาที่ขอจริงแล้ว (ไม่โชว์ order นอกช่วง)
- ผลลัพธ์บันทึกไว้ที่ `strategy/s20.12/excel/compare_s20_12.csv` (คอลัมน์: ..., **SIM_Lot**, SIM_P&L, SIM_Balance, ..., **MT5_Volume**, MT5_P&L, ...) — เทียบ lot ระหว่าง backtest กับของจริงได้ด้วย

### 3. สิ่งที่ backtest ยังขาด / ยังไม่ตรงกับ live 100% (ค้างอยู่)

ที่แก้ไปแล้ว (symbol, timezone, look-ahead bias, ความหน่วง 1 แท่ง, market entry, multi-position ต่อ TF, Shared TP, event-driven balance, hardcode skip-list ที่ขาด 20.12 ไปหลายจุด ฯลฯ) ทำให้ match rate ดีขึ้นมาก (จาก ~1% ขึ้นมา ~14% ล่าสุด) แต่ยังไม่ใช่ 100% เพราะ:

1. **Pattern detection บางจุดไม่ reproduce เหมือน live** — เจาะลึกแล้วพบสาเหตุย่อย 2 อย่าง:
   - **Bot-restart-boundary**: order ที่เกิดก่อน/หลัง bot restart คนละรอบ อาจใช้ order mechanism ต่างกัน (limit เก่า vs market ใหม่) ทำให้เทียบกันไม่ได้ — ไม่ใช่บั๊ก แค่ข้อมูลคาบเกี่ยวช่วงเปลี่ยนโค้ด
   - **MT5 historical data revision**: บาง order (entry/SL ตรงกับ backtest เป๊ะ แต่ TP ต่างกันมาก) เกิดจาก MT5/โบรกเกอร์แก้ไขราคาแท่งล่าสุดย้อนหลังเล็กน้อยหลังเกิดเหตุการณ์จริง — backtest (fetch ย้อนหลังทีหลัง) กับ live (real-time ตอนนั้น) เห็นข้อมูลไม่เหมือนกันเป๊ะ แก้ไม่ได้จากฝั่ง backtest
   - ยังมีบางเคสที่ไล่โค้ด static ทุกจุด (Quant Engine, Sweep/Trend Filter, Duplicate check, Adjacent-bar, SL Guard, Shared TP, PD Zone, News Filter) แล้วไม่พบสาเหตุชัดเจน — ต้องใช้ debug checkpoint เพิ่มเติมถ้าอยากขุดต่อ
2. **Compounding math ให้ตัวเลข $ ไม่สมจริง** — win rate สูง + R:R สูง (บาง setup RR 1:25, 1:60+) ทำให้ balance ทบต้นระเบิดเป็นหลักล้าน/สิบล้านได้ในไม่กี่วัน ตัวเลข % win/trade count เชื่อถือได้ แต่ตัวเลข $ ไม่ควรใช้อ้างอิงตรงๆ
3. **Spread เป็นค่าคงที่** — ดึงจาก `info.spread` ครั้งเดียวตอนเริ่มสคริปต์ ไม่ผันแปรตามสภาพตลาดจริงแบบ live (spread กว้าง/แคบตามช่วงเวลา/ข่าว)
4. **ไม่มี slippage/requote จำลอง** — คิดแค่ spread cost แต่ order จริงอาจ fill ที่ราคาต่างจากที่ตั้งใจได้อีกเล็กน้อยตอนตลาดผันผวน
5. **Match rate โดยรวมยังไม่เต็ม 100%** (ล่าสุด ~14%) — ดีขึ้นมากจาก fix หลายรอบ (โดยเฉพาะ Shared TP + PD Zone/Trend Recheck skip-list ที่เคยขาด 20.12) แต่ยังเหลือ pattern-detection-divergence (ข้อ 1) เป็นตัวจำกัดหลัก
